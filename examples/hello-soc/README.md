# Hello SOC — end-to-end bridge demonstration

This example walks **3 real atoms** through the complete FSM lifecycle
(`pending → in-progress → checked → verified → completed → closed`), with
the atom body generated by `atomic_dag.llm_bridge` on the `do` transition.

## What it demonstrates

- **The bridge in real use.** `llm_bridge.bridge_transition` calls a real
  `LLMProvider`, the response becomes the atom body, then
  `execute_transition` does its job (gate → FSM → atomic write → WAL).
- **Quality gate routing.** On `check`, the gate result decides routing —
  `verified` (pass) or `returned` (fail). The example atoms are built to
  pass the gate (gold/PQMS/VVV all above threshold), so the happy path
  goes through to `closed`.
- **Real atoms, recorded provider.** The atoms have valid frontmatter and
  travel through the real FSM. Only the LLM provider is swapped between
  manual and CI runs (ADR-009 D-bridge-5).

## How to run

### Default — recorded provider (CI-safe, no API key needed)

```bash
python examples/hello-soc/run_hello_soc.py
```

Loads `recorded/responses.json` and replays the responses in order. The
script copies the example into a fresh tmp directory and runs there, so
the in-repo source files stay untouched. The output prints the working
directory if you want to inspect the resulting atoms and WAL.

### Real LLM (manual only)

```bash
export ANTHROPIC_API_KEY=sk-...
pip install -e ".[llm]"
python examples/hello-soc/run_hello_soc.py --real
```

Calls Anthropic's API live. Output will vary per run — that is the point
of the recorded fixtures: deterministic CI from a non-deterministic source.

### In-place (mutates the repo example)

```bash
python examples/hello-soc/run_hello_soc.py --in-place
```

Runs against `examples/hello-soc/` directly instead of a tmp copy. The
atoms end up in `state: closed` with LLM-generated bodies, and a `.atomic-dag/wal.jsonl`
appears in the example directory. Useful for ad-hoc inspection; recover the
clean state with `git restore examples/hello-soc/`.

## Design decision (4.D)

Only the `do` transition uses `bridge_transition` (the LLM regenerates the
body). The follow-up transitions `check`, `next`, `last` are pure state
changes and use `execute_transition` directly — no LLM call needed for
state mutation. Per atom: **1 LLM call**. 3 atoms → 3 recorded responses.

This keeps the example honest: the LLM is a worker (writes content), the
Python assembler is the runtime (decides, validates, persists). Adding
LLM calls to `check/next/last` would conflate the layers (ADR-001).

## Files

```
examples/hello-soc/
├── README.md                  (this file)
├── atoms/
│   ├── 01-introduction.md     HELLO-001, deps []
│   ├── 02-method.md           HELLO-002, deps [HELLO-001]
│   └── 03-conclusion.md       HELLO-003, deps [HELLO-002]
├── recorded/
│   └── responses.json         3 entries — one per do-transition
└── run_hello_soc.py           workflow + RecordedProvider helper
```

The atoms form a linear DAG (`001 → 002 → 003`). The example processes
them in sorted order and walks each one through the full lifecycle
independently.
